UFS Pattern Generator — 架構報告

TC Flow → IR → Pattern (.py) ｜ 以 PF002_0098 Boot Stress Test 為例

這套系統把測試案例流程(TC)自動轉成可執行的 UFS 測試 Pattern(.py)。 每個產生的程式碼都紮根於兩個真實來源:程式碼知識圖譜(understand-anything) 與整合過的知識庫(llm-wiki)。所有 LLM 步驟用當前模型執行,無需 API key。

三個核心原則
  • 確定性 Python 只做可重現的工作:解析、查詢、組裝、驗證。
  • 當前模型 推理步驟(標註、生成、grounding)交給驅動的模型,不綁特定供應商。
  • 雙來源 grounding:code 答「怎麼實作」、wiki 答「規格/客戶約束」。

IR 實作細節

IR(Intermediate Representation)是 TC 與生成之間的橋樑。 為什麼需要它?整個 Pattern 一次丟給模型 → 粒度太粗、精度差;每個 step 獨立生成 → 丟失 phase 間依賴。 IR 用 dependency graph 描述 phase 順序與資料流,讓模型以 step 為單位、以 phase 為 context 生成。

IR 產生的 3 個階段(ir_generator/)

Stage 1 · parser.py規則式解析 markdown
## Phase / ## Loop / ### Step → 抽 SCSI CMD / UFS Query / opcode / idn / expected
→
Stage 2 · wiki_lookup.py查 ingested wiki
per-phase entities/concepts + 全域 conflicts.md
→
Stage 3 · LLM A標註資料流
phase inputs/outputs + edge data_flow
Stage 1/2 是確定性 Python;Stage 3 由當前模型做。最後 finalize-ir 把骨架 + 標註合併成 IR。

parser 抽取的內容(確定性,無 LLM)

項目來源 / 規則
Phase## Phase N / ## Phase F(regex [0-9A-Za-z]+,支援字母 phase)/ ## Loop
Step### Step X.Y / #### Step X.Y
SCSI / UFS / opcode / idnstep 區塊內的 **SCSI CMD** / **UFS QUERY** + 表格的 Opcode / Query Opcode / IDN
Loop 類型標題含 (N 次) → count;否則 condition;iterate_over → for_each
fail_conditionexpected 含 != / == 0x / Data Match → 自動加 fail_condition + on_fail: abort

IR JSON 結構(以 PF002_0098)

{
  "pattern_id": "PF002_0098",
  "phases": [
    { "phase_id": "phase_0", "type": "sequential",
      "outputs": ["boot_lun_id"],            // ← Stage 3 (LLM) 補
      "steps": [
        { "step_id": "step_0_4", "ufs_query": "WRITE ATTRIBUTE",
          "query_opcode": "0x04", "idn": "0x00 (bBootLunEn)",
          "expected": "QUERY RESPONSE Success" },
        { "step_id": "step_0_5", "expected": "bBootLunEn != 0x00",
          "fail_condition": "bBootLunEn == 0x00", "on_fail": "abort" }   // ← parser 自動偵測
      ] },
    { "phase_id": "loop_stress", "type": "loop",
      "loop_type": "count", "loop_count": 100,
      "inputs": ["boot_lun_id", "write_pattern"] }
  ],
  "dependency_graph": {                        // ← Stage 3 (LLM) 補
    "nodes": ["phase_0", "phase_1", "loop_stress"],
    "edges": [ { "from": "phase_0", "to": "phase_1",
                 "type": "sequential", "data_flow": ["boot_lun_id"] } ]
  }
}
Stage 1 產出 phases/steps(含 fail_condition);Stage 3 補上 inputs/outputs 與 dependency_graph.edges 的 data_flow。生成階段(④)再把它拓撲排序攤平成 step plan。

llm-wiki 衝突優先度

ingest 時,6 個知識源若對同一參數有不同值,依優先度解決。 這是 wiki grounding 能勝過 raw Spec 的關鍵,也是必須讀 ingested wiki(而非關鍵字搜尋)的原因。

兩條「刪除規則」(落敗的值會被移除)

規則衝突結果
Rule 1CustomerReq vs SpecCustomerReq 勝,Spec 值刪除

其他任何其他組合(如 Spec vs Script)兩者都保留並標註

實際解決的 2 個衝突(wiki/conflicts.md)

#規則覆蓋內容對 pattern 的影響
1CustomerReq > Spec WriteBooster 的 Attribute/Flag 操作:LUN 必須是 Normal、非 Boot、index 0–7,否則回 invalid INDEX。(Spec 原本無此限制 → 已刪除) 生成時要驗證 WB LUN：_assert_valid_wb_lun()
2UserPrompt > ModelDefault 未指定 LUN 時 = 容量最大的 Enabled LUN(ModelDefault 的 lun=0 → 已刪除) 不可寫死 lun=0:_select_default_lun()
為何重要:這兩條覆蓋只有讀 ingested wiki(conflicts.md + entities)才看得到。 若用 raw Spec 或關鍵字搜尋,會誤用 Spec 的寬鬆值 → 生成錯誤的 pattern。 (PF003 第一版用 grep 就犯了寫死 lun=0 的錯,改用 ingested wiki 後才修正。)

為什麼這樣設計

問題:能不能用 /understand-knowledge 把 code 和 wiki 合併成一張圖?
決定:不行。它會覆蓋 code graph、不建立跨來源連結、把 .py 當純文字。 所以「結合」改在生成階段做,兩個來源各自保持最擅長的形式。(方案 A)
問題:第一版用「key 比對」事先抓 grounding,結果如何?
決定:產生假陽性——名字子字串比對把 WRITE(10) 接到 write_attribute、 TEST UNIT READY 接到 is_tester_cmd。比查不到更危險(看起來成功卻是錯的)。 改成逐 step 生成 + 即時查圖譜,丟掉整個 key 機制。
問題:code grounding 為何要走圖譜而非 grep?
決定:圖譜給結構化定位(file:line + 呼叫鏈),比文字搜尋精確, 且能讀到真實原始碼切片。grep 容易因命名假設而漏(見下方 debug 案例)。
問題:wiki grounding 為何不能用關鍵字搜尋?
決定:關鍵字搜尋會繞過衝突解決。例如 PF003 的「預設 LUN」—— raw Spec 沒限制,但 UserPrompt 覆蓋成「容量最大的 LUN」;WriteBooster 的 LUN 限制來自 CustomerReq。 只有讀 conflicts.md + ingested 頁面才看得到這些覆蓋。
問題:為何拿掉 API key、用當前模型?
決定:讓系統可攜到任何 agent(如 Hermes),不綁 Claude Code、不需 anthropic SDK。 誰驅動就用誰的模型。runtime 只需 Python 3.10+ 與 repo 本身。

兩個被 debug 抓到的真實 bug

① 命名前綴假設

搜尋 START STOP UNIT 時假設類別叫 CmdSeq*, 但它其實叫 StartStopUnit(無前綴)→ 被 grep 過濾掉,留了 TODO。

修正:放寬搜尋(領域關鍵字 + 同時搜 pattern/ 真實用例), 並寫進規則「不要假設命名前綴」。

② Parser 靜默丟棄 Phase

PF003 的 ## Phase F 被丟掉——parser 的 regex 只配對數字 phase(Phase \d+),字母 phase 整個消失卻不報錯。

修正:regex 改 [0-9A-Za-z]+。 教訓:靜默丟棄最危險。

Debug 心法: validate 通過 ≠ 功能完整。一定要搭配看 provenance.json 與 # TODO human-confirm,才知道哪些是真正 grounded、哪些是待人工確認的洞。
