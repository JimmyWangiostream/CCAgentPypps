# 10.7.13 Data Out Transfer Rules

Data out transfer rules related with RTT have been defined for the following reasons:

• To have the same implementation of UFS data transfer mechanisms for data out transactions across various host and device vendors.

• To limit the number of outstanding RTTs sent by the device based on host capability.

RTT requests related to several commands and from any logical units may be sent in any order.

Examples of commands with data out transfer are: MODE SELECT (10), WRITE (6), WRITE (10), WRITE (16), FORMAT UNIT, SECURITY PROTOCOL OUT, etc.

The following rules are applicable for device in generating the RTT and the host in handling the RTT:

• **Rule 1** - The host shall send only one DATA OUT UPIU for each RTT received from the device.
  ○ Figure 10.7 shows an example of UTP traffic related to a write command processing: the host sends one and only one DATA OUT UPIU for each READY TO TRANSFER UPIU sent by the device.

[**Figure 10.7 — Example for Data Out Transfer Rule 1**

This is a sequence diagram showing communication between Host and Device:

1. Host sends COMMAND UPIU(Write) to Device with "Expected Data Transfer Length = 4MB"

2. Device sends RTT UPIU¹ to Host with "Data Buffer Offset = 00000h, Data Transfer Count = 48KB"

3. Host sends DATA OUT UPIU¹ to Device with "Data Buffer Offset = 00000h, Data Transfer Count = 48KB"

4. Device sends RTT UPIU² to Host with "Data Buffer Offset = 0C000h, Data Transfer Count = 32KB"

5. Host sends DATA OUT UPIU² to Device with "Data Buffer Offset = 0C000h, Data Transfer Count = 32KB"

6. Device sends RTT UPIU³ to Host with "Data Buffer Offset = 14000h, Data Transfer Count = 16KB"

7. Host sends DATA OUT UPIU³ to Device with "Data Buffer Offset = 14000h, Data Transfer Count = 16KB"]
