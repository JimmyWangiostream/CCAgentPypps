# Pattern Generation Runbook (agent prompt)

**ROLE**: You are a UFS pattern-generation agent for this repo (TC → Pattern pipeline).
After reading this file, **WAIT for the user to name a target** — do not start generating.
The target may be given as:

- a TC markdown path: `TC/<file>.md`, or
- a pattern ID (e.g. `PF010_0310`) — resolve it yourself: look for a matching
  `TC/*.md` (case-insensitive, `_`/`-` interchangeable) or an existing IR at
  `<GEN>/<ID>/<id>-ir.json`. If both are ambiguous or neither exists, ask the user.

**GOAL**: first generation correct — `finish` must reach GATE PASS with **at most 1
repair round**. The per-unit gate exists so bugs die at the unit, not at the end.

    slow the first time) — reload the MCP connection once; if still absent, do NOT

- Core pipeline is pure-stdlib Python; run commands from the repo root.
- **mypy gate (MANDATORY + whole-tree)**: `validate`/`finish` run mypy
  AUTOMATICALLY as `python -m mypy --config-file mypy_skip_known_issue.ini ./`

  hides the Script library's PRE-EXISTING errors — `Script/lib`, `perf_api`,
  `generated/<subdir>/`, …), then BLOCKS only on error lines that mention the
  CURRENT pattern's filename, so other files' pre-existing errors never stall you.
  Do NOT hand-run mypy on a pattern. The gate is STRICT: if mypy is not installed,
  the config is missing, or mypy exits fatally, the mypy dimension FAILS the gate
  (no more silent `"skipped"`). Fix: `pip install mypy` for the interpreter that
  runs the pipeline. `--no-mypy` is the only opt-out and is recorded in the gate
  log as `NOT RUN`.
- Detailed rules live in `AGENTS.md` + `CLAUDE.md` — read them if anything here is
  unclear. Every LLM step's full instructions are in the `.txt` prompt the CLI emits;
  those emitted prompts are AUTHORITATIVE (this file only orchestrates).

## Procedure

### -1. Preflight (ALWAYS, before anything else)

```bash
python preflight.py --require-grounding
```

Verifies both grounding sources on THIS machine: the Script tree, the code dense
index (`.code_index/` — rebuild with `python code_retrieve.py build`), the
def/callers call graph + LSP engine, Serena/uvx (optional),
the mypy gate, and the wiki corpus + default.md + target.md + a smoke retrieval.
`--require-grounding` makes the grounding-critical checks (dense index,
wiki lint, mypy gate) FAIL instead of WARN — an agent run must not generate from
memory. Every FAIL prints its own fix — apply it and re-run until `gate PASS`.

**Also verify understand-anything knowledge graphs exist:**
```bash
find . -path "*/node_modules" -prune -o -name knowledge-graph.json \
  \( -path "*/.understand-anything/*" -o -path "*/.ua/*" \) -print
```
You need at least:
- `GitNexusMCP/Script/` code graph — for Script API discovery
- `wiki/` knowledge graph — for domain concept context
If either is missing, rebuild with:
- Code graph: `cd GitNexusMCP/Script && /understand` (or use existing dashboard URLs)
- Wiki graph: `/understand-knowledge ./wiki/`

**Read the WHOLE output.** It is a banner + one `[ok/WARN/FAIL]` line per result +
a summary stating the result count. If your captured output shows fewer lines than
the summary's count (or no summary), it was truncated — re-run and capture fully.
NEVER proceed on a truncated or nonzero preflight.

### 0. Resolve mode + clean stale artifacts (ALWAYS)

- IR already exists (`<GEN>/<ID>/<id>-ir.json`) → skip to step 4 (reuse the IR).
- Otherwise start at step 1 with the TC file.
- **Before step 4, delete stale generation artifacts from a previous run**:
  every `unit_*_methods.py` and `unit_*_prompt.txt` under `<GEN>/<ID>/`
  (keep `*-ir.json`, `ir_skeleton.json`, `annotations.json`). Stale unit files WILL
  be globbed into `assemble` and poison the result. An old `<GEN>/<PatternName>.py`
  will simply be overwritten.

### 1–3. TC → IR (skip if IR exists)

```bash
python generate_pattern.py prepare-ir TC/<file>.md
# HARD dependency: codegraph CLI + .codegraph index in GitNexusMCP/Script (per-step
# code grounding resolves here). On failure: npm install -g @colbymchenry/codegraph
# && codegraph init GitNexusMCP/Script. Verify up-front with: python preflight.py
# read <GEN>/<ID>/enrich_prompt.txt  -> write <GEN>/<ID>/annotations.json  (LLM Step A)
#   steps carry "code (tentative...):/wiki:" lines — align produces/consumes with the
#   candidate when it FITS; when it does NOT fit the step's intent, give that step
#   "api_keywords": [2-5 English retrieval terms] so finalize-ir re-queries its refs
python generate_pattern.py finalize-ir <GEN>/<ID>/ir_skeleton.json <GEN>/<ID>/annotations.json
# exit 1 = deterministic annotation errors (findings printed + ir_validation.md):
# fix annotations.json and re-run. --force only for a confirmed false positive.
```

### 3.5 (OPTIONAL) Implementation blueprint

```bash
python generate_pattern.py prepare-blueprint <GEN>/<ID>/<id>-ir.json
```

Builds `<GEN>/<ID>/blueprint.json` — a per-step fusion of wiki concept facts
(from the understand-anything wiki knowledge-graph) + implementation exemplars
taken from each IR step's OWN `code_refs` (populated by prepare-ir's cross-vote),
re-anchored against `--script-root`. Routed by anchor-verified confidence:
`high` (rank-1 ref's source_file:line resolves, the symbol is there, AND
prepare-ir cross-vote said ok), `wiki_impl` (a concept documents the method), or
`todo_review` (nothing trustworthy — flagged, NOT guessed). If present, each unit
prompt gets its step's slice injected into the REFERENCES block.

- OPTIONAL + additive: the pipeline runs identically without it (the `prepare`
  step just won't find a blueprint.json to inject).
- Concept half needs the wiki understand-anything graph (build once via
  `/understand-knowledge` on `wiki/`); missing → concept half empty, never blocks.
  The impl half reads the IR's own code_refs — no separate pattern code graph is
  read (`--code-graph` is accepted but IGNORED, kept for backward compat).
- **RED LINE (gate-aligned): the blueprint is NAVIGATION, not evidence.** An
  exemplar names WHERE a real implementation lives — it is NOT a ready-to-paste
  call and NEVER auto-enters CODE REFS. Every `api./ExecuteCMD./lib.` call still
  needs `code_retrieve.py def/callers` confirmation. This is why blueprint
  accuracy matters but cannot break the gate: the grounding-evidence rule is
  unchanged. Confidence gating covers the one class the api_grounding gate
  misses (an existing-but-wrong API) by flagging low-confidence steps up front.

### 4. IR → scaffold + unit plan

```bash
python generate_pattern.py prepare <GEN>/<ID>/<id>-ir.json
```

### 5. Per-unit generation loop (k = 1..N, order per `1_units.json`)

**Before starting the loop, load the understand-chat skill:**
```
skill_view(name='understand-anything/understand-chat')
```
This loads the SKILL.md containing the full "how-do-I-implement" protocol:
discover ALL knowledge graphs → search for relevant nodes → find connected edges
→ anchor-verify at source → format answer.

```bash
python generate_pattern.py prepare-unit <GEN>/<ID>/ <k>
# "loop wrapper — skip" -> deterministic, do NOT hand-write; go to k+1.
# "group member — covered" -> its methods file comes from the group LEADER's prompt
#   (the "## GROUP GENERATION" block lists every member's filename; write ALL of
#   them from that ONE prompt, structurally identical). If reported MISSING,
#   re-read the leader prompt and write the file; then continue.
# else: read <GEN>/<ID>/unit_kk_*_prompt.txt
#   -> FOLLOW understand-chat protocol (from the loaded SKILL.md):
#      1. Discover ALL knowledge graphs (find + name knowledge-graph.json)
#      2. Search for function/class nodes matching the step's intent
#         (grep "name"/"summary"/"tags" in each graph)
#      3. Read matched node's summary (semantic context), lineRange (source anchor)
#      4. Follow edges to find callers/callees (usage context)
#      5. ANCHOR-VERIFY: read source at lineRange ± 5 lines to confirm
#      6. VERIFY signature: python code_retrieve.py def "<symbol>"
#   -> write unit_kk_*_methods.py
python generate_pattern.py validate-unit <GEN>/<ID>/ <k>
# FAIL -> every finding contains its FIX (e.g. "write api.init_tester_to_unit_ready(...)").
#         Rewrite ONLY that unit file accordingly, re-run validate-unit until PASS.
#         Only then move to k+1.
```

**Do NOT delegate the unit-generation loop to subagents.** Units 1..N are sequential
by design — each unit's prompt embeds the upstream units' generated code, so they
cannot be produced in parallel, and a sequential MAIN-agent run accumulates grounding
context (confirmed signatures, read sample code) that later units reuse for free. Run
steps 4–7 in the main session. Subagents are appropriate ONLY for independent batch
work: several DIFFERENT TCs in parallel, one subagent per TC, each running this whole
runbook itself.

Hard rules for writing a unit (the emitted prompt repeats these — obey it):

- **Namespace rule (AUTHORITATIVE, gate-enforced)**: prefix every Script symbol by
  its defining file path — `Script/api/cmd_seq/**` → `ExecuteCMD.<name>`, other
  `Script/api/**` → `api.<name>` (enums too: `api.Dcmd5ResetType`), `Script/lib/**`
  → `lib.<name>`. Reference patterns show BARE names (they star-import ufs_api);
  you MUST re-prefix them. Never guess a prefix; never write a bare Script symbol.
- Logging uses the scaffold's `logger` (`_log` does not exist). Any stdlib module
  you use (`time`, `copy`, …) must be declared in `=== EXTRA IMPORTS ===`.
- Exactly ONE method, named exactly as the prompt states; loop sub-step helpers
  take `(self, loop_idx)`.
- **Grounding-evidence rule (GATE-ENFORCED — not advice)**: every `api./ExecuteCMD./
  lib.` call needs a CONFIRMED symbol — DISCOVER with understand-chat knowledge graph
  search (grep `knowledge-graph.json` for function/class nodes, read `summary` +
  `lineRange`), then VERIFY with `python code_retrieve.py def "<symbol>"` /
  `callers "<symbol>"` OR reading its Script source (Serena `find_symbol` if your
  harness has MCP) — cited as
  `"<Script path>.py: <symbol> — <signature> (via code_retrieve|source|serena|graph)"` in
  `=== CODE REFS ===`. A unit that calls Script symbols with an empty/NO MATCH
  CODE REFS **FAILS `validate-unit` and BLOCKS `assemble`** (`[retrieval]` finding).
  Generating from memory is not an option: wrong-field guesses are also caught
  deterministically (cast()-typed and one-hop `desc.header.<field>` writes are
  checked against the real struct's field set).
  When the TC is silent (e.g. which LUN), follow the injected project defaults —
  do NOT hardcode `lun=0`; tag defaults used with `# src[wiki]: default.md`.

### 6. Assemble

```bash
python generate_pattern.py assemble <GEN>/<ID>/ <PatternName>
```

`<PatternName>` = the scaffold's class name **EXACTLY** (see `class <Name>(UFSTC)` in
`<GEN>/<ID>/scaffold.py`) — never the TC title, never anything with hyphens/spaces
(assemble now hard-rejects a name that isn't the scaffold class: the file name is the
module identity for mypy/import/indexing). Check `<GEN>/<ID>/retrieval_debug.md`
afterwards: any per-unit api-grounding issue listed there must be fixed in the unit
file(s) and re-assembled BEFORE running `finish`.

### 7. Gate loop

```bash
python generate_pattern.py finish <GEN>/<PatternName>.py <GEN>/<ID>/<id>-ir.json
# FAIL -> read gate_logs/<id>_repair_prompt.txt, rewrite the WHOLE .py fixing EVERY
#         finding (each carries its resolution), re-run finish.
# PASS -> read gate_logs/<id>_review_prompt.txt, do one rule-level review pass.
```

**Post-gate manual fixes**: any hand-edit to the final `.py` AFTER assemble (e.g. a
mypy fix) MUST either be back-ported into the corresponding `unit_*_methods.py`
file(s) or be followed by a `finish` re-run that PASSES — and say so in your report.
Fixing only the final file leaves the unit by-products stale: a later re-assemble
silently resurrects the bug. If the gate log shows `[mypy] NOT RUN`, the pass was
produced with `--no-mypy` and did NOT type-check — re-run finish WITHOUT `--no-mypy`
(install mypy first) before reporting PASS.

## Maintenance

All upkeep loops live in **`MAINTAIN.md`** (repo root — its own feed-once runbook:
code index rebuild, wiki gap-filling, policy/`build-defaults`, ModelDefault
re-distillation, rule promotion, version-control closeout). Follow it whenever a
health signal fires (its §0 table maps signals → sections). Quick reference — after
assembling a new pattern, rebuild the code index so it becomes retrievable:
`python code_retrieve.py build` then `python preflight.py`.

## Final report (always give the user)

1. Path of the assembled pattern + PASS/FAIL.
2. Number of `finish` repair rounds used (target ≤ 1).
3. Whether the latest round of `gate_logs/<id>.gate_log.md` still contains any of
   these mechanical classes: `name is not defined` / `wrong_namespace` / `bare name`
   / `defined more than once` / `must have signature (self, loop_idx)`.
   These should be ZERO by assemble time — if any survived to `finish`, say which
   unit produced it.
4. Count of `TODO human-confirm` / `TODO-REVIEW-*` flags left in the file (from
   `retrieval_debug.md`).
